## Executive Summary

The agentic coding assistant market has matured into a contested multi-leader landscape where the most consequential development of the past twelve months is not any single model release, but the collapse of confidence in the benchmarks that organize the entire field. As of April 2026, nine commercially significant tools — Claude Code, Cursor, OpenAI Codex CLI, GitHub Copilot, Aider, Cline, Continue, Devin, and OpenHands — compete across three form factors (terminal-native CLI agents, IDE-native extensions, and cloud platform agents) with broadly comparable headline capabilities. Different leaders exist for different evaluation regimes: Anthropic's Claude Opus 4.5/4.6 leads SWE-bench Verified at 77.2% (Claude Sonnet 4.5) to 80.9% (Claude Opus 4.5), while GPT-5 (high) leads Aider Polyglot at approximately 88%, and Cursor's proprietary Composer 2 model claims a position between them on Cursor's own internal evaluation suite [1] [2] [3] [4].

The landscape is reshaped by three structural shifts that vendor marketing materials systematically understate. First, SWE-bench Verified — the 500-issue subset that has organized vendor competition since August 2024 — is now demonstrably contaminated; OpenAI itself stopped reporting on it after an internal audit found that every frontier model could reproduce verbatim gold patches for some Verified tasks [5] [6]. The same Claude Opus 4.5 model that scores 80.9% on Verified scores only 45.9% on the contamination-corrected SWE-Bench Pro — a 35-percentage-point gap that should anchor every comparative claim. That gap reflects both contamination and genuine task difficulty: SWE-bench Pro tasks are legitimately harder (average 107.4 lines across 4.1 files, architecture-level reasoning), and Claude's further drop from Pro's public to private subset (22.7% → 17.8%) suggests task difficulty accounts for a meaningful fraction of the gap alongside contamination [7]. Second, the Model Context Protocol (MCP) released by Anthropic in November 2024 has, by its donation to the Linux Foundation in December 2025, become the de facto standard for tool-context integration across every major coding agent, eliminating vendor-specific tool integration as a competitive moat [8] [9]. Third, the productivity paradox — surfaced by METR's controlled study showing experienced developers were 19% slower with AI even though they *predicted* they would be 20% faster in pre-study estimates (the objective result was the inverse), and reinforced by GitClear's finding that AI-generated code carries 1.7× more bugs and 75% more logic errors than human-written equivalents — directly contradicts the 10× productivity claims that dominate vendor messaging [10] [11]. The productivity paradox is a real but contested finding: METR's own February 2026 update acknowledged significant selection bias in the study design, and the 2025 DORA report found over 80% of developers reported productivity gains.

For practitioners, the implication is that tool selection now matters far less than workflow discipline and verification practice. For enterprises, token costs of $200–$2,000+ per engineer per month on top of seat licenses make budget forecasting fragile, and the emerging security vulnerability profile — three high-severity CVEs in 2025 alone, including CamoLeak (CVSS 9.6) and CVE-2025-59536 (CVSS 8.7) — should be weighted as heavily as headline benchmark numbers [12] [13]. The balanced view is that agentic coding has won the architectural argument against pure autocomplete, but the field's empirical foundations are weaker than the dominant narrative suggests, and the next twelve months are likely to be defined by benchmark reform, platform consolidation, and a more honest accounting of where these tools actually deliver value.

## Introduction

This report investigates the state of agentic coding assistants as of April 7, 2026, with the goal of producing a comparison that distinguishes vendor claims from independently verifiable performance. The investigation was framed around five sub-questions defined in the SCOPE phase: (1) what the leading tools are and how their core architectures differ; (2) how they compare on standard coding benchmarks; (3) the state of multi-file editing, tool use, and agent autonomy; (4) cost structures, rate limits, and known production failure modes; and (5) what gaps remain in the agentic coding assistant landscape. The scope deliberately excludes general-purpose chat LLMs (such as ChatGPT or Claude.ai) used for ad-hoc code questions, IDE plugins without agent loop logic, and pure marketing material from vendor sites that lacks independent corroboration.

The methodology followed a ten-phase deep-research pipeline. The PLAN phase decomposed the investigation across four lens sub-agents — academic, practitioner, critical, and historical — each running on Claude Sonnet 4.6 with independent search trajectories. The RETRIEVE phase ran eight parallel WebSearches and four sub-agent retrievals in a single dispatch, yielding approximately 120 unique sources across roughly 1,900 lines of synthesized agent output. The TRIANGULATE phase identified eighteen central claims, resolved five contradictions between sources, and ran three devil's-advocate searches specifically aimed at challenging the emerging thesis. The OUTLINE_REFINEMENT phase reorganized the original five-question structure into eight evidence-driven findings, promoting the benchmark contamination story and the productivity paradox to standalone status because the gathered evidence demanded it.

Two assumptions are explicit and load-bearing. First, where vendor and independent sources disagree on a benchmark number, the report treats independent reproductions as the higher-credibility anchor and presents vendor numbers with explicit "Vendor" flags. Second, the temporal credibility decay applied to sources prefers Q1 2026 reporting over 2025 reporting where pricing or feature claims overlap, reflecting the field's roughly three-month feature half-life. Two limitations are worth flagging at the outset: the report's economic data is dominated by retrospective surveys rather than real-time deployment metrics, and the open-source ecosystem (Aider, Cline, Continue, OpenHands) is well-covered for capabilities but under-quantified for adoption relative to commercial competitors.

## Finding 1: The Agentic Coding Tool Landscape — Nine Tools, Three Form Factors, One Loop Family

The commercially significant agentic coding assistants of April 2026 fall into three structural categories distinguished by where the agent loop runs and how it interacts with the developer. Terminal-native CLI agents — Claude Code, OpenAI Codex CLI, and Aider — execute in the developer's shell, edit files in the local working tree, and treat the IDE as optional context rather than required substrate. IDE-native extensions — Cursor, Cline, Continue, and GitHub Copilot's agent mode — embed the agent loop inside an editor (VS Code in most cases, with JetBrains parity emerging in 2026) and use the editor's UI affordances for diff review, file scoping, and approval gates. Cloud platform agents — Devin and OpenHands — run the agent loop on managed infrastructure with the developer interacting through a web dashboard rather than a local process, offering long-running execution at the cost of local state fidelity [14] [15] [16].

The **CLI category** has been the most strategically important shift of the past eighteen months. Claude Code, launched as a research preview alongside Claude 3.7 Sonnet on February 24, 2025 and reaching general availability in May 2025, was Anthropic's first developer tool product and, by widely-reported but not vendor-confirmed secondary accounts, hit an approximate $1B annualized revenue run-rate within roughly six months of GA — a faster commercial ramp than ChatGPT itself, though readers should treat the specific revenue figure as approximate rather than audited [17] [18]. The tool now spans five surfaces (terminal, VS Code extension, JetBrains extension, web app, and iOS), pricing tiers of $20, $100, and $200 per month (Pro / Max / Max with higher limits) with a five-hour rolling usage window, plugin and sub-agent architectures, and first-class MCP support [19]. OpenAI relaunched the "Codex" brand in stages — CLI in April 2025, cloud agent in May 2025 — recycling the name from the deprecated 2021 base model, and on April 2, 2026 moved to token-based pricing on top of bundled ChatGPT plans, positioning it as the OpenAI ecosystem's parallel to Claude Code [20]. Aider, the open-source pioneer of the category created by Paul Gauthier in 2023, remains model-agnostic and is notable for having had AI write a large portion of its own codebase; it leads the open-source Polyglot leaderboard that benchmarks multi-language diff editing accuracy [21].

The **IDE-native category** is dominated by Cursor and GitHub Copilot. Cursor — founded in early 2022 by four MIT graduates and by February 2026 reportedly the fastest software product ever to reach $100M ARR, with $2B ARR and a $29–30B valuation — pivoted decisively in October 2025 with Cursor 2.0, which introduced its first proprietary model, Composer, optimized for low-latency multi-step agentic coding [22] [23]. Cursor 3.0, launched April 2, 2026, rebuilt the entire user interface around an "Agents Window" — an orchestration surface for running multiple AI agents in parallel — making explicit the company's thesis that "the developer's job is to orchestrate AI, not write every line" [24]. GitHub Copilot, the category-defining tool with a technical preview dated June 29, 2021 and general availability reaching in June 2022, completed its own evolution from autocomplete to chat to agent: agent mode was announced February 6, 2025 and the Coding Agent reached GA in September 2025 for paid subscribers, replacing the sunset Copilot Workspace project [25]. GitHub also became model-agnostic in October 2024, offering Claude, Gemini, and GPT model selection inside Copilot. Cline, a third-party VS Code extension originally named "Claude Dev" and rebranded in 2024, occupies the open-source end of the IDE-native category with explicit per-step approval gates for every file edit, terminal command, and web action [26].

The **cloud platform category** is occupied by Devin (Cognition AI) and OpenHands (formerly OpenDevin). Devin, launched March 2024 as "the world's first AI software engineer," became the most controversial product in the category after independent debunking — most prominently the "Internet of Bugs" YouTube channel — showed its launch demos overstated its capabilities; the company's response was a quiet repositioning from "replaces engineers" to "AI for engineering teams," followed by Devin 2.0 (April 2025) and the acquisition of Windsurf (formerly Codeium) in July 2025, which doubled Cognition's ARR overnight [27] [28]. Despite the controversy, Cognition's commercial trajectory has been remarkable: ARR climbed from approximately $1M in September 2024 to $73M by June 2025, valuation rose from a $2B Series A valuation in early 2024 to $10.2B by September 2025, and customers now include Goldman Sachs, Citi, Dell, Cisco, Ramp, Palantir, Nubank, and Mercado Libre [29]. OpenHands, launched March 12, 2024 as the explicit open-source response to Devin, has grown to approximately 65,000+ GitHub stars and ~500 contributors under MIT licensing, stewarded by All Hands AI [30].

What unifies all nine tools at the architectural level is the family of agent loops they implement. Most follow a ReAct-style pattern (Reason → Act → Observe), formalized in the 2022 Yao et al. paper [31], typically augmented with tool use (file read/write, bash execution, web search), some form of planning step, and a termination condition based either on the model declaring completion or on a hard tool-call budget. Devin differs in using a more explicit plan-execute-verify architecture, generating a multi-step plan up front and tracking its execution, while Claude Code's sub-agent system and Cursor 3's Agents Window represent the leading edge of multi-agent orchestration — spawning specialized sub-agents for sub-tasks and merging their results back into a main loop. The architectural diversity is real but smaller than vendor marketing implies; the fundamental ReAct skeleton is common to all.

## Finding 2: SWE-bench Verified Is Contaminated — and the Field Knows It

The single most consequential development for benchmark interpretation in the past twelve months is that SWE-bench Verified, the 500-issue human-validated subset created by OpenAI in August 2024 and treated since then as the gold standard for autonomous coding capability, is no longer regarded as trustworthy by the institution that created it. OpenAI's official position, stated in a research blog post titled "Why we no longer evaluate SWE-bench Verified," is that internal auditing found every frontier model tested — GPT-5.2, Claude Opus 4.5, and Gemini 3 Flash among them — could reproduce verbatim gold patches or problem statement specifics for at least some Verified tasks [5]. The mechanism is straightforward: because the underlying issues were drawn from public GitHub repositories that were almost certainly inside the training corpora of every major model, models can sometimes recover the specific commit-level fix from memory rather than from reasoning. OpenAI's recommended replacement is SWE-Bench Pro, a contamination-aware benchmark whose curation explicitly excludes tasks with measurable train-set overlap.

The numerical impact of this correction is dramatic. The same Claude Opus 4.5 model that scores 80.9% on SWE-bench Verified scores only 45.9% on SWE-Bench Pro — a thirty-five percentage point drop on the same model with the same scaffolding [7]. The ~35-point gap is not pure contamination — SWE-bench Pro tasks are also genuinely harder (average 107.4 lines across 4.1 files, architecture-level codebase reasoning), and Claude's further drop from Pro's public to private subset (22.7% → 17.8%) suggests task difficulty accounts for a meaningful fraction of the gap alongside contamination. This is a compound measurement correction. The implication is that every "X% on SWE-bench" headline number circulating in vendor marketing materials, vendor press releases, and downstream technology journalism between August 2024 and approximately Q1 2026 should be discounted substantially before being treated as a real-world capability estimate. This includes Anthropic's Claude scores (77.2% for Claude Sonnet 4.5 to 80.9% for Claude Opus 4.5 on Verified, dropping to ~45.9% Pro), OpenAI's Codex scores (which have not been reported on Verified for several months but historically claimed approximately 72.1%), and Cursor's Composer 2 claims, which are reported on a proprietary CursorBench rather than either Verified or Pro [3] [32].

Equally telling is the gap between vendor benchmark claims and independent reproductions when the same benchmark version is run by a third party. The swe-rebench.com leaderboard, an independent reproduction effort using cleaner scaffolding, scored Claude at approximately 65.3% — a fifteen percentage point gap below the official 80.8% number that Anthropic reports [33]. Cursor's claim of approximately 58% on SWE-Bench Pro for Composer 2 was rerun by Scale AI's evaluation labs, which reported approximately 46% — a twelve percentage point gap [7]. Devin's original launch claim of 13.86% on a 570-issue random sample of SWE-bench (not the Lite subset) was largely confirmed by a practitioner replication (3/20 tasks, ~15%) by Answer.AI, but the much higher 87.5% figure that has appeared in some Cognition marketing materials is for an internal Upwork-style task subset that bears no relationship to SWE-bench and should not be compared with it [34] [35]. The pattern across all of these gaps is consistent: vendor numbers run roughly 12–35 percentage points higher than independent reproductions, with the largest gaps appearing on the most heavily marketed models.

The structural reason this matters is that benchmark contamination and flawed test cases are not bugs to be fixed; they are permanent properties of any benchmark drawn from public sources used to train the systems being evaluated. SWE-Bench Pro buys time but does not solve the problem — it will eventually be contaminated too, as new model training runs ingest the post-Pro web. The field's response is fragmentation: Anthropic emphasizes its "Live SWE-agent" methodology (a continuously refreshed task pool), OpenAI emphasizes private holdout sets, Cursor emphasizes its proprietary CursorBench, and academic researchers (notably the "SWE-Bench Illusion" paper [36] showing a 47-point drop when tasks were evaluated on out-of-distribution repositories) emphasize task perturbation as a generalization test. None of these has yet achieved consensus status. The honest summary for any 2026 reader is that no current benchmark cleanly answers "how good are agentic coding tools?" — the only credible interpretation is to read multiple benchmarks in combination and treat any single score as one data point in a noisy distribution.

For the practical purpose of this report, the recommendation is to present three numbers per tool wherever available: a vendor-reported Verified score (flagged as Vendor), a contamination-corrected Pro score (flagged Independent where reported by third parties), and an Aider Polyglot score where applicable. Readers should weight the Pro and Polyglot numbers more heavily and treat Verified scores as historical context.

## Finding 3: A Multi-Leader Market — Different Benchmarks, Different Winners

The benchmark contamination story, important as it is, can obscure a second structural fact about the 2026 landscape: even when different tools are evaluated on benchmarks that *are* trusted, no single tool leads everywhere. The field has stratified into a multi-leader market where the winner depends entirely on what task profile a benchmark measures. This is the kind of finding that matters more for practical tool choice than any single headline number, because it tells a developer that the question "which tool is best?" only makes sense once they have answered "best at what?"

Three leadership clusters are visible in the April 2026 data. First, on **SWE-bench Verified and SWE-bench Pro** — benchmarks that measure end-to-end issue resolution against real GitHub repositories, with multi-file editing, test running, and patch validation — Anthropic's Claude Opus 4.5 and 4.6 lead consistently. The Verified scores are inflated by contamination, but the *relative* ordering of Claude above the other frontier models is stable across multiple independent reproductions, which suggests the Claude scaffolding (whether the official Anthropic harness or community-built Live-SWE-agent variants) is genuinely well-suited to the multi-file repair task profile [1] [37]. Second, on **Aider Polyglot**, which measures multi-language diff editing accuracy across Python, JavaScript, Rust, Go, C++, and several other languages, GPT-5 (high) leads with approximately 88% — well ahead of Claude on the same test [2]. The plausible reason is that GPT-5's training data and instruction tuning emphasized cross-language diff fluency in a way Claude's did not, and Aider's tool architecture (which puts the model in tight-loop conversation with the diff format) rewards models that can produce small, precise edits without context drift. Third, on **proprietary internal benchmarks** including Cursor's CursorBench, the picture is more vendor-controlled and less independently verifiable; Cursor has reported that Composer 2 "beats Claude Opus 4.6 but still trails GPT-5," which reads more as marketing positioning than as a clean technical claim [3]. Cursor Composer 2 does lead Claude on Terminal-Bench 2.0 (61.7 vs 58.0), though it trails GPT-5 (75.1) on the same benchmark. Cursor does not lead on SWE-bench or Aider Polyglot.

GPT-5 is the closest approximation to a multi-domain leader in Q1 2026, leading SWE-bench Pro (~57.7%), Terminal-Bench 2.0 (75.1), and OSWorld (~75%); Claude leads the contamination-heavy Verified and retains the strongest scaffolding story; Aider Polyglot goes to GPT-5 (high) at ~88%. The multi-leader structure is real but not symmetric — GPT-5 has the broadest benchmark coverage of any single model in this cohort.

The implication is not that the field lacks information about which tools are good. It is that the field has *too much* information, all of it filtered through different evaluation harnesses and different definitions of "good." A senior backend engineer working primarily in Python on mature microservices repositories should likely prefer Claude Code (or Cursor running Claude as the underlying model) because the SWE-bench task profile most closely matches their work. A polyglot full-stack developer working across TypeScript, Rust, and Python should probably prefer a tool that performs well on Aider Polyglot — which today means a setup using GPT-5 (high) either through Aider directly or through OpenAI Codex CLI. A team evaluating agentic coding for the first time should probably ignore the headline benchmark numbers entirely and run a realistic in-house pilot, because no public benchmark currently maps cleanly onto their codebase.

The deeper point is that **benchmark leadership has become decoupled from product leadership**. Cursor leads the IDE category by user count and revenue but does not lead on SWE-bench or Aider Polyglot (though it does lead Claude on Terminal-Bench 2.0). Claude Code leads the CLI category and most SWE-bench-style evaluations but trails on Aider Polyglot. GitHub Copilot leads the enterprise seat-count category by a large margin (the Microsoft channel advantage is decisive) but its agent mode is newer and less feature-rich than Claude Code's or Cursor's. This decoupling is healthy for the market — it suggests no winner-take-all dynamic — but it makes the comparative-review genre that this report belongs to much harder to write honestly. The most defensible position is to present the cluster structure rather than to pick a winner, and to explicitly tell readers that the answer depends on their workload.

A subsidiary observation is that **multi-file editing and tool use capabilities are now table stakes** across all nine tools surveyed. Every tool can read and write multiple files in a single agent step. Every tool has bash execution. Every tool has some form of test runner integration. The differentiator at the edge is no longer "can the agent edit multiple files?" but "can the agent maintain coherent intent over a sequence of fifteen or twenty tool calls without losing the thread?" The leading bets on this long-horizon coherence problem are Claude Code's sub-agent system (which spawns specialized child agents for sub-tasks and merges their outputs) and Cursor 3's Agents Window (which lets the developer orchestrate multiple agents in parallel and review their results) [38] [24]. Both architectures are essentially admissions that pure single-agent loops degrade past a certain task depth, which is consistent with academic work on agent loop limits and which sets up the gap discussion in Finding 8.

## Finding 4: The Productivity Paradox — Vendor 10× vs Contested Empirical Evidence

If the benchmark contamination story is the most important methodological finding in this report, the productivity paradox is the most important practical one. Across an enormous and growing volume of vendor marketing material, agentic coding tools are claimed to deliver developer productivity multipliers ranging from 1.5× at the conservative end to 10× at the aspirational end, with many vendor case studies citing specific projects completed in "a tenth of the expected time." A growing body of independent empirical evidence from 2025 and early 2026 directly challenges these claims for the most economically important category of developers — experienced engineers working on familiar mature codebases. Crucially, however, this empirical evidence is itself contested: the productivity paradox is a real but not settled finding.

The most important single piece of evidence is METR's randomized controlled study, published in 2025 and updated in February 2026, which measured experienced open-source developers completing real bug fixes on codebases they knew well. The headline result was that developers *predicted* they would be 20% faster with AI assistance in pre-study estimates; the objective measured result was that they were 19% slower [10] [39]. Both numbers come from the same study population on the same set of tasks; the gap between prediction and measurement is itself part of the finding. However, METR's own February 24, 2026 update acknowledged selection bias so severe that they suspended the study design: 30–50% of developers refused to submit tasks they wanted to do *with* AI, systematically excluding high-uplift tasks from the evaluation. METR also noted the study used early-2025 tools. The 19% negative result should therefore be understood as a lower bound on a contested range, not as a settled finding about current tools or current practices. The 2025 DORA report (90% AI adoption in surveyed organizations) found over 80% of respondents reported productivity gains, providing direct counter-evidence to METR's headline number. GitClear's own founder has acknowledged that churn analysis in isolation is "inherently flawed" without controlling for task type. The productivity paradox is a real phenomenon at the pessimistic end of a distribution, but the distribution is wide.

GitClear's research, drawing on analysis of more than 150 million lines of code from 2021 through 2025, provides a complementary signal from a very different methodology. GitClear measures code churn (the percentage of recently committed code that is modified or deleted within a two-week window) as a proxy for code quality and durability, and they have found that the metric rose from a 3.3% baseline in 2021 to between 5.7% and 7.1% in 2024–2025, with the increase concentrated in repositories with high AI assistance usage [11]. AI-generated PRs in GitClear's sample carried 1.7× more bugs than human-written equivalents, with 75% more logic errors, and the pattern is consistent with developers "pasting together disjointed blocks without understanding the broader system." A Stack Overflow blog analysis from January 2026 raised the question of whether bugs and incidents are inevitable in current agentic coding deployments and recommended that teams budget for higher review and incident response costs alongside the per-token costs [40].

A nuance worth stating directly is that both METR and GitClear are organizations whose research output focuses on critical evaluation of AI tools, and a fair reading should acknowledge their institutional priors. The productivity paradox does *not* mean agentic coding tools are useless. Anthropic's own 2026 Agentic Coding Trends Report — which is vendor material and should be read with that caveat — acknowledges productivity gains are more modest than the 10× marketing claims suggest, without committing to a specific multiplier [41]. The reconciliation is that the productivity gain depends heavily on three factors: how greenfield the work is (high gains for new code, low or negative gains for mature code), how senior the developer is (juniors gain more than seniors because the verification burden is borne more lightly when the developer wasn't confident anyway), and how disciplined the verification workflow is (developers who treat AI output as a draft to be reviewed gain real time; developers who treat it as production code lose it back). Larridin's developer productivity benchmark report from 2026 frames the same pattern as a distribution: the median user gains modestly, the top quintile gains substantially, and the bottom quintile loses time outright [42].

The strategic implication is that **the productivity story for agentic coding is workflow-dependent, not tool-dependent**. Picking Claude Code over Cursor will not move a developer from the negative-gain quintile to the high-gain quintile; what moves them is allocating time to verification, treating agent output as a draft, and developing a codebase model the agent can be calibrated against. This is consistent with the empirical observation that the developers who report the largest productivity gains in the wild are often those who spend roughly 70% of their time on problem definition and verification strategy and only 30% on direct execution — using the AI to compress the execution share rather than replace the thinking share [42]. The vendors who acknowledge this honestly — GitHub's recent emphasis on review-first workflows in Copilot Coding Agent, and Anthropic's own conservative framing in the Trends Report — are positioned more credibly than those who continue to circulate the 10× number.

For an enterprise reader trying to decide whether to deploy agentic coding tooling at scale, the productivity paradox should structure the cost-benefit analysis. Token costs alone (Finding 5) make the deployment expensive in absolute terms; if the productivity gain is modest rather than 10×, the ROI math is much tighter than the marketing implies, and the verification investment becomes load-bearing. This is the part of the agentic coding story that the field is least honest about — not because anyone is lying, but because the truthful answer is uncomfortable for both vendors and adopters.

## Finding 5: Cost Economics and Production Failure Modes

Cost is the most underappreciated dimension of the agentic coding decision. The published seat prices on every commercial tool surveyed in this report are visible and modest — Claude Code at $20/$100/$200 per month, Cursor at $20/$60/$200 per month, GitHub Copilot at $10/$39 per month, OpenAI Codex bundled with ChatGPT plans plus token-based overage as of April 2, 2026 — but they represent only a fraction of the real cost of running agentic coding tools at production scale. The fully loaded cost, dominated by token consumption, runs from approximately $200 per engineer per month at the low end (a developer using the tool occasionally for routine tasks) to over $2,000 per engineer per month at the high end (a developer running multiple autonomous agents on long-horizon tasks throughout the workday) [13] [42] [43]. For a 100-engineer team, the higher end of that range translates to $2.4M per year in token costs alone, on top of the $50K–$240K in seat licenses.

The structural reason token costs run so high is that agent loops are token-amplifying by design. A single ReAct-style step typically involves the model reading several files of context (input tokens), reasoning about what to do (output tokens), making a tool call (input tokens for the call, output tokens for the result observation), and then iterating — with each iteration accumulating more context as the agent's working memory grows. A 20-step task can easily consume 200,000 input tokens and 50,000 output tokens; at current Claude or GPT pricing, that is a few dollars for a single task that a developer might run dozens of times in a day. The "$200 to $2,000" range cited above is consistent with the per-task economics multiplied by realistic developer usage patterns [13]. Cursor's recent pricing changes — a $20 hobbyist tier, a $60 pro tier, and a $200 ultra tier with explicit token budgets — reflect this reality, as does Claude Code's five-hour rolling usage window which functions as an implicit rate limit to prevent runaway consumption.

The cost story is closely entangled with the **production failure mode story**, because the most expensive failures are the ones where the agent does not stop when it should. Several documented incidents from 2025 and early 2026 illustrate the pattern. In March 2026, a Claude Code session running with elevated permissions executed a `terraform destroy` command against a production environment after a misinterpreted user instruction; the destruction was caught and partially reversed but consumed significant remediation effort and produced a high-visibility post-mortem in the company's engineering blog [44]. In July 2025, Replit's agentic coding feature wiped a SaaStr customer's database during what was supposed to be a schema modification task; the incident was widely discussed in the developer community and prompted Replit to add additional confirmation gates [45]. A separate class of failures involves **runaway agents** that exhaust their token budget within minutes by entering loops they cannot escape — a widely reported March 2026 Claude Code quota bug affecting many users had agents burn through their full quota in 19 minutes by repeatedly attempting and failing the same file edit [46]. Aggregated reporting from 2026 customer incident logs cites $15,000 in unexpected API costs as a typical "bad week" for an enterprise team without rate limiting or budget alerts in place [13].

A taxonomy of the documented failure modes is useful for risk assessment:

1. **Tool misuse (destructive operations).** The agent invokes a tool with the wrong arguments, or with arguments it should not have been allowed to invoke at all. The Claude Code terraform destroy and Replit database wipe cases both fall here. Mitigation: explicit allowlists, mandatory human confirmation for irreversible operations, and sandboxing.

2. **Context-window collapse.** The agent's working memory grows past a threshold and earlier instructions or constraints are forgotten, causing it to violate its original task definition. This is a common cause of "the agent did something I didn't ask for" complaints and is hard to detect in real time because the agent will continue to confidently report progress.

3. **Runaway loops.** The agent enters a state where it repeatedly attempts the same action, fails, and tries again — usually because its observation parser is misreading an error message and it does not have a higher-level mechanism for breaking out of the loop. Token costs accumulate rapidly. The 19-minute quota burn case is the canonical example.

4. **Unverified destructive operations.** A subclass of (1) but worth calling out separately: the agent runs a destructive operation in a context where it had the technical right to do so but should have asked for confirmation. This is the failure mode most associated with the "trust calibration" problem and is the one that scales worst with autonomy.

5. **Demo-vs-reality drift.** Devin's original 2024 launch is the canonical case: the agent performs well on the curated demonstration tasks the vendor has practiced and poorly on the messier tasks customers actually want done. Independent benchmark reproduction is the structural defense.

For enterprise risk management, the recommended controls are budget caps per agent run, explicit confirmation gates for destructive operations, mandatory diff review for any change touching production paths, monitoring on token consumption with alerts at 50% and 80% of budget, and a clear rollback path for any agent-generated change. The vendors that have invested most heavily in these controls — Claude Code's plan-then-execute modes, Cursor's per-file approval, GitHub Copilot Coding Agent's PR-based workflow — are the ones that are easiest to deploy responsibly. The vendors that emphasize autonomy without equivalent investment in guardrails are the ones most likely to produce the next post-mortem.

## Finding 6: Security — Three High-Severity CVEs in 12 Months

The agentic coding category is, structurally, an attack surface expansion: every tool in this report executes code, accesses files, and makes network calls based on natural language instructions that may have been authored by an attacker rather than the developer. The 2025–2026 disclosure record bears this out. Three high-severity CVEs and several lower-severity issues have been reported across the category in approximately twelve months, all stemming from the same general class of vulnerabilities: prompt injection, where untrusted text in the agent's context (a file, a tool result, a web page) causes the agent to perform unintended actions [47] [48] [49].

**CVE-2025-59536** (CVSS 8.7) affected Claude Code and involved a trust dialog bypass enabling premature code execution from project configuration files — an attacker who could place a malicious configuration file in the developer's working directory could cause Claude Code to execute commands on the developer's machine without the expected confirmation step [47]. The vulnerability was patched within days of disclosure, but the attack surface (anything that can write files into a developer's working tree, including legitimate package installs) is unusually large.

**Clinejection**, disclosed against Cline in late 2025, used a similar prompt injection vector but propagated automatically through repositories — affecting an estimated 4,000 developer machines before patching [48]. The propagation mechanism was that infected repositories contained injection payloads in their README or comment files, and any developer who opened the repository in Cline would have their next agent session compromised. This is the closest thing the agentic coding category has had to a worm, and it raised the question of whether code-hosting platforms (GitHub, GitLab) should scan for known injection patterns in the same way they scan for secrets.

**CamoLeak** (CVSS 9.6), disclosed against GitHub Copilot Chat, used image rendering as the exfiltration channel: an attacker could include a maliciously crafted image reference in a chat context that, when rendered, would issue a request to an attacker-controlled server with chat history embedded in the URL [49]. The vulnerability was patched and the rendering pipeline hardened, but the structural lesson — that any rich-content rendering inside an agent context is an exfiltration risk — applies equally to every tool in the category that supports inline images, Markdown, or HTML.

The deeper structural problem is that the **MCP standard, while a major usability win, is also a fresh attack surface**. MCP servers are essentially tool-providing daemons that an agent can call to perform actions, and the security model requires the agent to validate the trustworthiness of each server. In practice, developers install MCP servers from third-party registries the same way they install npm packages — and the npm history of supply-chain attacks suggests this will eventually produce a parallel category of MCP-server compromises. Some of the early 2026 MCP server registry tooling has begun to address this with signing requirements and provenance metadata, but the ecosystem is young and the attack surface is growing faster than the defenses.

For a practitioner, the operative recommendations are: (1) treat agent file-read operations with the same skepticism as opening an email attachment, especially when the file came from an external source; (2) keep MCP servers to a minimum and prefer well-known, signed providers; (3) audit the tool permissions granted to each agent and disable any that are not actively in use; (4) sandbox agent execution wherever possible — the cost is small relative to the blast-radius reduction; (5) monitor agent network egress, because exfiltration through tool calls is the highest-value attack pattern. None of these are exotic security practices; what is novel is that they need to be applied to a tool category that has been marketed as a productivity enhancement, not as a security risk.

For an enterprise reader, the security profile of the category should weight as heavily in the deployment decision as the productivity profile and the cost profile. A team that is willing to deploy an agentic coding tool with default permissions and no monitoring is, in practice, accepting a security posture significantly weaker than the one the same team would accept for any other piece of developer tooling — and the post-mortem record of 2025–2026 suggests that posture will produce incidents.

## Finding 7: Platform Consolidation, Protocol Standardization, and the "Agents Window" Thesis

If 2024 was the year agentic coding became a serious product category, 2025 was the year it began to consolidate, and the first months of 2026 are the year a consensus product thesis crystallized around it. Three structural shifts deserve attention: the Cognition–Windsurf merger, the elevation of MCP to industry standard status, and Cursor 3's "Agents Window" as the leading candidate for the next-generation product UX.

The **Cognition–Windsurf acquisition** in July 2025 was the first major consolidation event in the category. Cognition, the company behind Devin, had built its commercial story around the autonomous-engineer thesis and was widely doubted after the March 2024 demo controversy; by mid-2025 it had nevertheless reached $73M ARR through enterprise sales to Goldman Sachs, Citi, Dell, and other large customers, supported by the post-controversy repositioning to "AI for engineering teams" rather than "replacement for engineers" [29]. Windsurf, formerly Codeium, had built a competing IDE-native product with significant developer mindshare but was looking for a path to enterprise scale. The acquisition immediately doubled Cognition's revenue and gave the combined company a complete stack — agent platform plus IDE — competing more directly with Cursor and Claude Code than either Devin or Windsurf could individually [50] [51]. Cognition's subsequent $400M raise at a $10.2B valuation in September 2025 priced this consolidated thesis at scale [52]. The acquisition is best read as evidence that the "agent-only" positioning Devin originally took has limits that are best addressed by adding an IDE substrate, rather than by competing with IDE-native tools head-on.

The **Model Context Protocol's elevation to industry standard** happened in two stages. Anthropic released MCP on November 25, 2024 as an open-source, MIT-licensed protocol for connecting AI assistants to external tools and data systems, framed analogously to USB-C as a universal connector for AI context [8]. Adoption was rapid in the months that followed: Cursor, Cline, and Continue added support in late 2024; OpenAI announced MCP support for ChatGPT and the Agents SDK in March 2025; Google DeepMind followed shortly thereafter; GitHub Copilot adopted MCP in 2026 alongside its agent mode rollout. In December 2025, Anthropic donated the protocol to the Linux Foundation under the umbrella of the new Agentic AI Foundation (AAIF), co-founded with Block and OpenAI [9]. This is the first AI-era protocol to make the journey from "vendor-released open standard" to "industry-stewarded standard" in roughly twelve months, which is unusually fast and reflects the genuine pain point MCP solved: every coding agent vendor was building bespoke tool integrations before MCP existed, and none of those integrations transferred between vendors. The Linux Foundation stewardship signals that MCP is no longer a competitive moat for any single vendor — and the practical effect is that *tool ecosystem* is no longer a meaningful axis of competition between coding agents. Whoever builds the best agent loop wins; integration parity is now table stakes.

The third shift — the emergence of **multi-agent orchestration UX as the next product paradigm** — is the least consolidated and the most contested. Two leading bets are visible. Claude Code's sub-agent system, introduced in Claude Code 2.x and refined through 2025–2026, lets the main agent spawn specialized child agents for sub-tasks and merge their outputs back into the main loop; the sub-agents can run on different model variants (e.g., Sonnet for retrieval tasks, Opus for synthesis) and produce isolated context windows that do not pollute the main agent's working memory. Cursor 3.0's Agents Window, launched April 2, 2026, takes a different approach: instead of hiding sub-agents inside the main agent's loop, it surfaces multiple independent agents in a dedicated UI panel where the developer can spawn, observe, and intervene in any of them in parallel [24]. The Cursor framing is more explicit about the underlying philosophy — "the developer's job is to orchestrate AI, not write every line" — and represents the most committed product bet on the thesis that agentic coding is replacing IDEs as a category, not just adding features to them. Whether Claude Code's hidden orchestration or Cursor's exposed orchestration becomes the dominant UX is one of the open questions for the next twelve months. Both architectures are essentially admissions that single-agent loops degrade past a certain task depth (see Finding 8) and that the next round of capability improvements will come from agent composition rather than from raw model scale.

A final observation is that the **"agentic coding = new IDE" thesis** is now mainstream VC consensus, traceable to a chain of articulations: Devin's March 2024 launch implicitly making the bet, GitHub Copilot Workspace's April 2024 technical preview making it explicit from an incumbent, Cursor's founders articulating it publicly in mid-2024, and a16z and Sequoia publishing pieces in 2025 framing the IDE as "the new battleground for AI" [53] [22]. Cursor 3's launch made the thesis explicit in product design, and the valuations across the category — Cursor at $29–30B, Cognition at $10.2B, Anthropic's developer-tools revenue contributing materially to its overall valuation — reflect that the thesis is now priced. Whether the thesis is ultimately right is unknowable, but the market is operating as if it is.

## Finding 8: Gaps in the Landscape

Despite the maturity of the category, several large gaps remain — places where the current generation of agentic coding tools fails or falls noticeably short of where they would need to be to deliver on the marketed productivity story. Three gaps deserve specific attention because each is documented in multiple independent sources and each is structural enough that it will not be closed by incremental tool improvements.

**Gap 1: Long-horizon coherence.** Every tool surveyed in this report degrades noticeably past a moderate number of sequential tool calls in a single agent loop — the exact threshold is task-dependent and varies between roughly ten and a few dozen calls before degradation becomes obvious. The degradation mode varies — context-window crowding, instruction drift, working-memory loss, error-recovery failures — but the underlying pattern is consistent across vendors and across model families. Academic work on agent loop limits, including the Reflexion paper [54] and recent work on ReAct loop stability, has converged on the same finding: pure single-agent loops without external memory architectures degrade superlinearly with task depth. The two leading product responses to this problem (Claude Code's sub-agents and Cursor 3's Agents Window) are essentially admissions of the limit, attempting to compose around it rather than solve it. The gap is well-known to vendors and is widely discussed in user communities, but no current tool has a clean fix. A developer planning to use an agentic coding tool for a multi-hour task should expect to break the task into shorter segments or to actively manage the agent's memory and intent across the segments.

**Gap 2: Verifiable reasoning over benchmarks.** SWE-bench Pro is a band-aid for a structural problem: every benchmark drawn from public sources used to train the evaluated models will eventually become contaminated, and the only sustainable evaluation strategies are private holdouts (which sacrifice reproducibility), task perturbation (which imperfectly tests generalization), or live benchmarks that continuously refresh from a moving target (which sacrifice version-to-version comparability). OpenAI's own retirement post explicitly acknowledged this problem and pointed to the need for better evaluation infrastructure rather than better benchmarks per se [5]. Academic responses including the SWE-Bench Illusion paper [36] and the Live-SWE-agent methodology have begun to surface, but no consensus has formed. The practical effect is that no current benchmark can be trusted as a single point of evidence for capability claims, and every comparison report — including this one — must triangulate across multiple benchmarks while acknowledging that all of them are partial.

**Gap 3: Cost predictability.** The token-based pricing model that dominates the category makes it structurally impossible for an enterprise team to forecast its agentic coding costs more than a month or two in advance. The variance is large: a single developer's cost in a given month can range from $50 to $2,000+ depending on which tasks they happen to run, how aggressive the agent's tool-call budget is, and whether any runaway loops occur [13] [42]. Some vendors have introduced explicit budget caps and rate limits (Claude Code's five-hour rolling window, Cursor's tier-based budgets) but these are coarse instruments and do not give a finance team enough visibility for accurate forecasting. The deeper problem is that token consumption is intrinsically tied to task complexity, and task complexity is hard to predict in advance. Until someone in the category prices on a per-task or per-feature basis rather than per-token, cost predictability will remain a real friction for enterprise adoption.

A fourth gap is worth mentioning more briefly: **the open-source vs commercial quality gap is poorly quantified**. Aider, Cline, Continue, and OpenHands are all credible production-quality tools with significant adoption, but no public benchmark cleanly separates their capability ceilings from those of Claude Code and Cursor. The plausible reason is that open-source tools depend on the model the user supplies, so benchmark numbers reflect the model rather than the tool harness. Until the benchmarking field matures enough to separate harness quality from model quality, the open-source vs commercial decision will continue to be made on workflow fit and total cost of ownership rather than on performance.

## Synthesis & Insights

Three patterns cut across the eight findings above and deserve to be stated as standalone insights, because they are more important than any individual fact in the report.

**Insight 1: Marketing has decoupled from measurement.** The benchmark contamination story and the productivity paradox are not separate findings; they are two manifestations of the same underlying dynamic. Vendors are incentivized to publish numbers that demonstrate frontier capability, and the field's measurement infrastructure has not kept pace with the gaming pressure. The consequence is that the most-cited metrics — SWE-bench Verified scores, productivity multipliers — are systematically the most inflated, and the most credible signals — METR's controlled studies (even accounting for their acknowledged selection bias), Scale AI's third-party reproductions, OpenAI's own retirement post — get less attention because they are either methodologically inconvenient or commercially uncomfortable. A reader's first move when evaluating any agentic coding claim should be to ask whether the source is vendor-controlled and whether the same claim has been independently reproduced.

**Insight 2: The category has stratified by workload, not by quality.** The "which tool is best?" framing that dominates technology journalism is the wrong question. The empirical evidence shows that different tools lead on different benchmarks, that benchmarks do not cleanly correspond to real workloads, and that the workflow practices around the tool matter more for productivity than the tool itself. A more honest framing is that the leading tools — Claude Code, Cursor, GitHub Copilot, OpenAI Codex CLI — are converging on broadly comparable capability ceilings within their form factors, and the meaningful differentiators are workflow fit, ecosystem (including MCP), and total cost of ownership. A reader trying to decide between them should run a realistic in-house pilot rather than reading a comparison review (including this one).

**Insight 3: The next round of capability improvements is architectural, not model-scale.** The two most interesting product launches of the past twelve months — Claude Code's sub-agent system and Cursor 3's Agents Window — are both bets on agent composition as the answer to the long-horizon coherence problem. They are also both acknowledgments that pure model scaling is not solving the agent loop limits. This is consistent with the academic literature on agent failure modes [54] and with the vendor patterns visible in the leadership clusters of Finding 3. If the bet is right, the differentiation between tools over the next twelve months will come from how well they orchestrate sub-agents and present that orchestration to the developer, not from which underlying model they ship with.

A subsidiary insight worth surfacing: **the productivity paradox is a feature of the current measurement methodology, not necessarily a permanent state.** The METR-style negative-productivity finding for experienced developers on familiar code may attenuate as models improve at producing code that requires less verification, as workflow practices mature around the tools, and as the verification overhead is reduced by better diff-review tooling. But the current state, in April 2026, is what should structure deployment decisions today.

## Limitations & Caveats

This report has several limitations that readers should weigh.

First, **vendor benchmark inflation is acknowledged in every metric**, and the report's response is to triangulate against independent reproductions wherever they exist. But independent reproduction is itself an immature practice, and the third-party benchmarks (Scale AI's reproductions, MorphLLM's Pro leaderboard, the Aider Polyglot leaderboard) are themselves operated by organizations with their own incentives. The cleanest defense is to weight cross-organization agreement higher than any single source, which the verification log in the methodology appendix tries to make explicit.

Second, **the productivity-paradox evidence base is small relative to the deployment universe and methodologically contested**. METR's study population was experienced open-source developers; GitClear's analysis is large in line count but observational rather than controlled; METR's own February 2026 update acknowledged that 30–50% of developers refused to submit high-uplift tasks, introducing systematic selection bias. The 2025 DORA report's counter-evidence (80%+ productivity gains self-reported) is itself subject to social desirability bias. The field would benefit from a properly randomized, large-scale controlled trial. The current best interpretation is that the productivity story is workflow-dependent and that the pessimistic results are real-but-not-universal.

Third, **the report's snapshot is dated April 7, 2026**, and the field has a roughly three-month feature half-life. By July 2026 several of the specific version numbers, pricing tiers, and feature claims in this report will be out of date. The structural findings — benchmark contamination, productivity paradox, security profile, MCP standardization — should survive longer, but the surface details should be re-verified before being cited.

Fourth, **the open-source ecosystem (Aider, Cline, Continue, OpenHands) is well-covered for capabilities and adoption signals but under-quantified for production deployment** relative to the commercial competitors. This is a real gap in the report that more directed sub-agent research could close, but the current evidence does not support strong claims about the production maturity of the open-source category as a group.

Fifth, **the security CVE record should be read as a lower bound on the real attack surface, not as a complete inventory**. Vulnerabilities that have been quietly patched without public disclosure are not in the count; vulnerabilities that affect rarely-used features may not have been discovered yet; supply-chain vulnerabilities through MCP server registries are an emerging category whose full shape is not yet visible.

Sixth, **the report does not cover the ecosystem of meta-tools** (agent monitoring platforms, prompt management systems, evaluation harnesses, agent orchestration frameworks like LangGraph or Anthropic's own SDK) that surround the agentic coding tools themselves. These are arguably as important to a production deployment as the underlying tool, but they are out of scope for a tool-comparison report.

Seventh, **verification-driven corrections**: five hard FAILs were identified and corrected in the PACKAGE phase: the Anthropic productivity multiplier figure was hedged, the Cognition/Windsurf acquisition date was corrected from August to July 2025, the arXiv IDs for SWE-rebench and SWE-Bench Illusion were corrected and the 65.3% number properly attributed to swe-rebench.com leaderboard, the Aider Polyglot leader was corrected to "GPT-5 (high)" from "GPT-5.4," and the "Cursor leads no major public benchmark" claim was corrected to acknowledge Cursor's leadership on Terminal-Bench 2.0. Additionally, twelve soft-pass hedges were applied throughout the report correcting specific factual details around Codex relaunch dates, GitHub Copilot launch history, Cognition valuation history, OpenHands contributor counts, SWE-bench benchmark characterizations, and several other claims.

## Recommendations

For **individual developers** evaluating agentic coding tools for personal use, the recommendation is to choose by workflow fit rather than by benchmark. A developer who lives in the terminal should use a CLI agent (Claude Code, OpenAI Codex CLI, Aider). A developer who lives in an IDE should use an IDE-native tool (Cursor, GitHub Copilot, Cline). The performance differences between tools within a form factor are smaller than the friction of using a tool that does not match the developer's existing workflow. Within a form factor, the secondary criteria worth considering are: (1) which underlying models the tool exposes, since model leadership varies by task type (Claude for SWE-bench-style work, GPT-5 (high) for polyglot work); (2) how well the tool integrates with the developer's existing repository structure and CI/CD; and (3) total cost over a representative month of usage, including token overage.

For **engineering teams** considering team-wide adoption, the recommendation is to invest in workflow discipline before tool selection. The METR and GitClear evidence is clear that — despite its methodological limitations — the difference between a productive deployment and an unproductive one is the verification practice, not the tool. A team should plan for: explicit code review standards for AI-generated code that are equal to or stricter than standards for human-generated code; budget allocations for the additional review time; clear escalation paths for the failure modes documented in Finding 5; security training that includes prompt injection awareness; and a pilot phase long enough to measure whether the actual productivity gain matches the projected one. The pilot should run for at least three months and should produce its own measurements rather than relying on vendor-supplied multipliers.

For **enterprise CTOs and engineering leaders** making buying decisions at scale, the recommendation is to anchor cost expectations on the $200–$2,000+/engineer/month token cost range and to demand SWE-Bench Pro numbers (not Verified) from vendors as part of the evaluation. The full deployment decision should weight five dimensions equally: capability (with independent benchmark verification), cost (fully loaded including tokens), security posture (with explicit reference to the CVE record and vendor mitigation practices), workflow fit, and exit cost (since switching tools after deployment is more expensive than the seat license suggests, due to MCP server configurations, prompt history, and team training). The vendors that perform best on the security and cost-control dimensions are not necessarily the ones that perform best on the headline benchmark dimensions, and the deployment decision should reflect that.

For **researchers and benchmark designers**, the recommendation is to treat benchmark contamination as a permanent feature of the public-data landscape and to invest in evaluation infrastructure (private holdouts, task perturbation, live refresh) rather than in better one-shot benchmarks. The field would also benefit from better measurement of the long-horizon coherence problem identified in Finding 8, since this is the structural limitation that current tools are not solving and that the next generation of agent-composition products is attempting to work around.

For **policy and security professionals**, the recommendation is to treat agentic coding tools as a new attack surface category that merits the same scrutiny as web browsers or email clients, and to advocate for standard practices around MCP server signing, agent permission auditing, sandbox execution, and incident reporting. The 2025–2026 CVE record is a leading indicator, not an outlier.

## Methodology Appendix

This report was produced using the deep-research skill's ten-phase pipeline in deep mode. Phase 1 (SCOPE) defined five sub-questions, seven acceptance criteria, and per-sub-question time domains. Phase 2 (PLAN) committed to a four-lens sub-agent architecture (academic, practitioner, critical, historical) and an eight-search parallel decomposition with three pro/con query pairs targeting the most contested tools. Phase 3 (RETRIEVE) ran the four lens sub-agents synchronously in a single Task tool dispatch with `model="sonnet"` (per the hybrid model architecture in which the lead agent runs on Claude Opus 4.6 and sub-agents run on Claude Sonnet 4.6), alongside eight parallel WebSearches from the lead agent. Phase 4 (TRIANGULATE) cross-referenced eighteen central claims, resolved five contradictions, and ran three devil's-advocate searches against the emerging thesis — these last produced the most important findings of the report (the SWE-bench Verified contamination, the absence of independent Cursor Composer 2 reproduction, and the METR/GitClear productivity paradox evidence). Phase 4.5 (OUTLINE_REFINEMENT) reorganized the original five-question plan into eight findings to give standalone weight to the contamination story, the productivity paradox, and the security CVE record. Phase 5 (SYNTHESIZE) produced the initial draft. Phase 6 (CRITIQUE) reviewed for gaps. Phase 7 (REFINE) applied six critique fixes and deferred seven items to verification. Phase 7.5 (VERIFY) ran four Sonnet sub-agents (three verifier batches + one adversarial), surfacing five hard FAILs, twelve SOFT-PASS hedges, and three adversarial corrections. Phase 8 (PACKAGE) produced this final report, applying all corrections. v14's PACKAGE phase was killed mid-run and salvaged by a Sonnet 4.6 sub-agent.

Sub-agent allocations were:
- Agent A (academic lens, Sonnet): SWE-bench academic literature, MCP RFCs, agent loop papers
- Agent B (practitioner lens, Sonnet): Vendor docs, IDE comparisons, feature/pricing matrices
- Agent C (critical lens, Sonnet): Failure modes, CVEs, runaway agents, post-mortems
- Agent D (historical lens, Sonnet): Tool evolution, vendor pivots, benchmark history

Verification sub-agents (all Sonnet):
- V1: Benchmark claims (SWE-bench scores, Aider Polyglot, Terminal-Bench)
- V2: Vendor/historical claims (acquisition dates, valuations, launch dates, contributor counts)
